@workspace 請幫我修復「促銷活動管理」中，頁籤 (Tabs) 數字連動錯誤的 Bug。

目前的狀況：點擊不同頁籤進行資料篩選時，「全部」這個頁籤上面的數字徽章 (Badge) 會跟著變成當前篩選後的筆數，這是不對的。
修復要求：
請找到綁定給「全部」頁籤的數字變數。將它從「篩選後的資料長度 (例如 filteredList.length)」改為綁定「最原始、未經篩選的總資料筆數 (例如 totalData.length 或是由 API 返回的 totalCount)」。確保無論切換到哪個頁籤，「全部」上面的數字都固定顯示資料庫裡真正的總筆數。您可以保留極少數（例如 5 筆）的資料維持在「待審核」或「已拒絕」狀態即可，剩下的 90% 以上全數設為通過。


@workspace 請幫我修改「右側詳細資訊面板」底部按鈕區塊的條件渲染 (Conditional Rendering) 邏輯。

目前的狀況：「核准上架」與「退回」這兩個按鈕出現在太多不該出現的頁面（例如：商品總覽、已上架等）。
修復要求：
請在這兩個按鈕的外層容器（或按鈕本身）加上條件判斷（例如 v-if 或 && 運算子）。**必須且只有**在滿足以下四個情境之一時，才顯示這組審核按鈕：
1. 當前位於「商品審核中心」的【待審核】列表
2. 當前位於「商品審核中心」的【重新申請審核】列表
3. 當前位於「促銷活動管理」的【待審核】列表
4. 當前位於「促銷活動管理」的【重新申請審核】列表

如果是從「商品總覽」、「進行中」、「已拒絕」、「庫存監控」等其他任何地方打開這個面板，請將底部的審核按鈕區塊完全隱藏。可以透過傳遞 props (例如 isReviewMode=true) 或是檢查當前的路由/Tab狀態來實作這個判斷。
